Network Architecture and Synaptic Connectivity

The network topology uses a two-dimensional grid arrangement matching the spatial structure of the input images. For the current implementation, a 16 by 16 grid provides 256 neurons corresponding to 256 pixels. Each neuron connects to its spatial neighbors within a radius of 3 pixels, creating local connectivity similar to lateral connections in visual cortex.

Synaptic weights determine the strength of connection between neurons. In biological systems, these weights change through learning and experience. For this implementation, weights are calculated offline using a Hebbian learning principle. Neurons corresponding to pixels that frequently appear together in training images develop strong mutual connections. This creates an associative memory where the complete pattern becomes a stable attractor state in the network dynamics.

Figure 6: Visualization of synaptic weight matrix

The weight calculation process begins with training fragments. I manually created 20 cropped portions of the target image, each showing different regions with 40 to 60 percent of the complete pattern. For each fragment, a simple Hebbian rule strengthens connections between active neurons. If neurons i and j are both active in a fragment, the weight connecting them increases by a small amount. After processing all fragments, the weight matrix is normalized and thresholded to eliminate weak connections.

This offline training produces a 256 by 256 synaptic weight matrix requiring 65,536 weight values. With 8-bit precision per weight, the matrix consumes 64 kilobytes of storage. The SDRAM provides ample space for this data along with neuron state variables.

During operation, the synaptic computation pipeline reads the current spike state of all neurons, multiplies each spike by the corresponding synaptic weight, and sums these contributions to calculate total input current for the target neuron. This sum becomes the input to the Izhikevich differential equations for that timestep.

The hardware implementation relies on a specific state machine within the `synapse_core.v` module to manage the multiply-accumulate process. Because the Cyclone IV has limited multipliers (only 56 embedded 9-bit elements), we cannot compute all inputs in parallel. Instead, for each target neuron update, the pipeline enters a `FETCH_SPIKE_VECTOR` state, reading the 256-bit spike history register from the previous 100ms timeframe. It then transitions to `ACCUMULATE`, iterating through a counter from 0 to 255.

A simple boolean AND gate checks the specific bit in the spike vector corresponding to the current counter index. If the bit is `0`, the pipeline asserts a `nop` (no operation) signal. If the bit is `1`, it triggers a read request to the weight matrix stored in SDRAM at `base_addr + (target_neuron_id * 256) + source_neuron_id`. This event-driven gating is physical, not simulated; the `mac_enable` signal on the DSP block stays low unless a spike is present, preventing dynamic power switching in the adder logic.

Homeostatic Stability of Electrical Impulses and Overall Regulation

Spiking neural networks can exhibit instability without proper regulation. Positive feedback through strong recurrent connections can cause runaway excitation where activity escalates uncontrollably. Biological networks prevent this through inhibitory interneurons and neuromodulatory systems that adjust excitability based on overall activity levels.

This implementation includes a global modulator neuron operating in parallel with the main neuronal sheet. The modulator monitors average firing rate across the network by counting total spikes per timestep. When activity exceeds a threshold of 80 spikes per millisecond, indicating excessive excitation, the modulator reduces bias current to all neurons. When activity falls below 20 spikes per millisecond, the modulator increases bias current to maintain ongoing dynamics.

This homeostatic control mechanism consumes approximately 980 logic elements as mentioned earlier and operates on a 10 millisecond timescale. The modulator updates its bias adjustment every 10 milliseconds based on accumulated spike counts, preventing rapid oscillations while maintaining responsiveness to sustained activity changes.

Testing shows this mechanism successfully prevents runaway excitation during network operation. Initial tests without the modulator resulted in either quiescent networks with no activity or explosive activity that saturated all neurons within 100 milliseconds. Adding the modulator allows the network to maintain stable dynamics with average activity levels between 40 and 60 spikes per millisecond.

The Implementation Issue Currently Being Resolved

At the time of writing, the complete NonPU design is implemented in Verilog and passes Analysis & Synthesis in Intel Quartus Prime Lite. The architecture described throughout this document has been validated through extensive module-level simulation where individual components including the Izhikevich pipeline, synaptic weight lookup, homeostatic gain controller, and VGA timing generator have been tested in isolation using ModelSim test benches. Edge case simulations confirm that the neuron dynamics exhibit correct behavior across boundary conditions including zero input current, maximum firing rates, and recovery from saturation states. The superneuron state machine transitions correctly between loading, inference, and idle modes in simulation. However, the full place-and-route Fitter stage for the AC415 target board is still failing due to timing constraints during the final routing phase.

The specific issue lies in simultaneous timing closure across multiple clock domains. The SDRAM interface operating at 133 MHz, the neuronal pipeline at 100 MHz, and the VGA pixel clock at 25.175 MHz create crossing boundaries that the current pin assignment and floorplan cannot satisfy within the Cyclone IV routing fabric. The Fitter reports negative slack on several critical paths between the memory controller and the computation core, indicating that signals arrive after the clock edge when they should arrive before. This timing violation prevents generation of a stable bitstream that would run correctly on physical hardware. I am actively debugging these compilation errors through iterative floorplanning adjustments, SDC constraint relaxation on non-critical paths, and potential architectural modifications to add pipeline registers at clock domain boundaries. The Efficiency and Event-Driven Architecture of Non-PU

One of the central motivations for the NonPU architecture is event-driven efficiency: the idea that computation and power draw should scale with real informational content in the input, not with a fixed matrix size. The current hardware bring-up state does not yet allow me to publish final watt-meter readings from the AC415 board, but the measurement methodology and comparison framework are already defined.

In the planned setup, power measures will be taken using a digital multimeter or inline power monitor connected directly to the AC415 FPGA supply rails. Three operating conditions define the test protocol beginning with Configured Idle where the bitstream is loaded and clocks are running but no external input activity or spike stimulation occurs. Moderate Activity involves input patterns with sparse spikes or low-contrast structure while High Activity introduces heavily corrupted patterns or dense random input that drives widespread spiking. For each case, the NonPU serves as a comparison against a conventional fixed-topology feedforward artificial neural network implemented on the same FPGA or a small GPU. The key quantity of interest is not raw speed but whether power consumption in the spiking system visibly tracks input activity. A mostly black image should produce close-to-idle power draw whereas a dense or noisy image should draw more current as more neurons spike and more synapses are exercised.

Exact watt values and scaling factors are intentionally omitted here, because they would be speculative until the bitstream compiles and the design runs stably on real hardware. Published GPU board powers (for example, 75 watts for an entry-level card) provide an approximate reference for the orders of magnitude involved, but the true value of this project is demonstrating the shape of the power curve: flat for matrix engines, activity-dependent for spiking hardware.

activitymonitor.v

The neuron sheet produces spikes. This module converts that stream into structured information about regional activity. Each tile in the 16×16 grid has a counter. When spike_valid fires, the module translates neuron index to tile position and increments that tile's counter. At frame_tick, the monitor scans all tiles, compares counts to threshold, and emits events for overactive or underactive regions. Counters reset. This is where spatial awareness exists.

fb_ram.v

Dual-port RAM. One side connects to spike2letter, which writes pixel values as neurons update. The other connects to display controller or testbench. Each memory entry holds one 8-bit pixel. Address equals neuron index. Neuron 0 is pixel 0. For a 16×16 sheet, depth is 256. The sheet writes new activity while another process reads the current image simultaneously. No arbitration required. True dual-port.

neuronalpipes.v

This module is one neuron in silicon. v_in and u_in represent membrane voltage and recovery variable from the previous timestep. I_in is total current from image stimulus and synaptic inputs. The core implements Izhikevich equations using fixed-point arithmetic. When voltage crosses threshold, spike_out fires for one clock cycle, voltage resets, and recovery adjusts. This block does not classify. Unlike dense matrix multiplication where every weight is computed every cycle, synaptic operations here are sparse and event-gated—only active connections contribute. It solves discrete differential equations in hardware.

sheet.v

The FPGA does not contain 256 separate neuron blocks. This module uses one neuronalpipes core and time-divides it across a 16×16 grid. Memory arrays store voltage, recovery, and input current for each neuron. A counter selects one neuron per cycle, reads its state, feeds it through the core, and writes updated values back. When a neuron fires, spike_valid asserts with the neuron index. When the counter wraps from neuron 255 to 0, frame_tick pulses. One physical core multiplexed over 256 logical neurons.

spike2letter.v

Humans cannot read spike trains. This module translates neuron activity to visible pixels. For each neuron update, it receives spike_valid, neuron_idx, and voltage v_in. If the neuron spiked, write 255 to framebuffer at that neuron's address. If not, rescale voltage to 8-bit grayscale and write that. Output signals fb_we, fb_addr, and fb_data connect directly to framebuffer RAM. In hardware, these values become monitor pixels. In simulation, they form an image of emergent network patterns.

superneuron.v

This is not another neuron. This is a control plane above the sheet. It maintains one bias value per tile in memory. When activitymonitor reports a tile firing excessively, this module reduces that tile's bias. Underactive tiles get increased bias. The internal variable G represents global gain, adapting based on overall activity. tile_id_query from the sheet indicates which tile the current neuron belongs to. The module looks up that tile's bias and outputs it as bias_out. The sheet adds this to neuron input current. Spikes shape biases. Biases shape spikes. Closed loop homeostasis.

All modules are synthesized using Intel Quartus Prime Lite Edition and target the Cyclone IV EP4CE15 FPGA. The complete source code repository includes additional documentation, testbenches, and synthesis scripts. For detailed implementation information, refer to the GitHub repository.

Neuronal Computation Pipeline Architecture

The neuronalpipes module implements the core neuronal computation engine that updates individual neuron states according to the Izhikevich spiking neuron model. This mathematical framework balances biological realism against computational efficiency, capturing essential dynamics of cortical neurons including spike generation, refractory periods, and diverse firing patterns while requiring only basic arithmetic operations implementable in digital logic.

Each neuron maintains two state variables encoded as sixteen-bit fixed-point numbers. The voltage variable represents membrane potential, rising in response to synaptic inputs and resetting abruptly when spike threshold is crossed. The recovery variable provides adaptation and implements refractory dynamics that prevent immediate re-firing after a spike. This pipeline implements the coupled differential equations defined in Izhikevich’s 'Simple Model of Spiking Neurons" paper.
Specifically, we solve for the membrane potential or "v" and the membrane recovery variable "u". Where the original model assumes continuous-time floating point evaluation, our architecture derives a discrete-time, time-multiplexed solution using 16-bit fixed-point arithmetic, allowing a single physical logic core to solve the dynamics for thousands of virtual neurons sequentially. These operations map naturally onto hardware arithmetic units within the FPGA fabric.

While I initially thought that this would be impossible with a Cyclone IV, but given that this Master Board has a 32MB SDRAM capable of storing states of more than 8 million neuronal weights, this became possible, Then I initiated this novel approach. Unlike a fully parallel implementation where every neuron has its own physical circuit, the NonPU acts as a high-speed pipeline. The FPGA logic core functions as a single physical engine that iterates through millions of computerized neurons stored in SDRAM. It fetches the state membrane variables v and u, applies the Izhikevich fixed-point update equations, and writes the new state back to memory. This cycle repeats 10 million times per second, creating the emergent behavior of a massive network using minimal silicon area.

The pipeline structure permits the same physical arithmetic hardware to be time-multiplexed across numerous virtual neurons. Rather than instantiating dedicated logic for each neuron in the simulation, the system loads state variables from memory, performs the update computation, and writes modified state back to memory in a cyclical fashion. This approach trades update frequency for neuron capacity, enabling simulations of far more neurons than would fit if each required dedicated hardware. The critical constraint becomes memory bandwidth rather than logic element availability.

For a million-neuron simulation, the computational requirements can be precisely calculated. Each neuron requires approximately ten clock cycles to fetch state from SDRAM, execute arithmetic operations, and write results back to memory. Multiplying one million neurons by ten cycles yields ten million cycles required per complete network update. With a fifty-megahertz clock providing fifty million cycles per second, the system achieves five complete updates per second. This five-hertz update rate represents ten-fold slower than biological neuronal networks operating in the gamma frequency range of thirty to eighty hertz, yet remains perfectly adequate for many neuromorphic applications and represents remarkable achievement for fifty-dollar hardware.

Global modulation from the superneuron module enters the pipeline through multiplication of synaptic inputs by the gain factor. This occurs before the inputs accumulate into the voltage variable, effectively scaling network sensitivity. Implementation in fixed-point arithmetic requires careful attention to numerical precision and dynamic range to prevent overflow or underflow conditions. The sixteen-bit representation with appropriate binary point placement provides sufficient range to capture voltage dynamics while maintaining adequate precision for stable numerical integration.

Spike generation occurs through simple threshold comparison. When voltage exceeds a predetermined level, the neuron enters the spike state for a single cycle, during which it broadcasts its spike to connected neurons via the synaptic network. The voltage then undergoes an abrupt reset to a hyperpolarized value while the recovery variable increments, implementing the refractory dynamics that ensure spikes remain discrete events rather than continuous high-voltage states.

Encoder Implementation Workflow

The encoding workflow begins with video ingest where source content transfers from external storage into the FPGA processing environment. For initial development and testing, transmission occurs via UART serial connection from a host computer, with the PC reading the source video file and streaming pixel data sequentially to the FPGA. This approach avoids the complexity of implementing video capture hardware while enabling controlled testing with known reference content. The transmission rate need not match real-time video rates since encoding operates offline with quality prioritized over speed.

As pixels arrive at the master board, they enter the differential analysis pipeline where current pixels compare against corresponding pixels from the previous frame retrieved from SDRAM. The master board transmits both pixel values to slave boards configured as difference computation engines. The slave boards execute simple subtraction operations and return magnitude results indicating the degree of change. When these difference values exceed predetermined thresholds, the system recognizes significant change events worthy of inclusion in the compressed representation.

The master board maintains responsibility for constructing the compressed representation in the SD card. Each significant change event generates a data packet encoding the temporal position, spatial location, and magnitude of modification. These packets accumulate in the SD card storage organized sequentially to permit efficient sequential reading during playback. The packet format balances compactness against ease of parsing, with fixed-size fields enabling rapid seeking to specific temporal positions without requiring variable-length record handling.

Key frame detection operates through accumulation of total change magnitude across each complete frame. A running counter increments as each pixel difference is computed and thresholded. When the frame completes, the system compares the total accumulated change against a threshold representing some percentage of the full frame size. If total change exceeds this threshold, the system designates the frame as a key frame requiring complete storage. The complete frame data is serialized and written to the SD card with appropriate markers indicating its role as a reference frame rather than a sparse differential packet.

The temporal extent of processing may span many hours or even days for complex video content. This extended timeline is not a deficiency but rather enables the sophisticated analysis necessary for optimal compression. During this period, the system can perform multiple passes over the source material if beneficial, first identifying key frames and then optimizing the differential encoding between them. The biological analogy to memory consolidation during sleep proves apt since the process reorganizes information for efficient storage and subsequent rapid recall.

Storage organization in the SD card follows a simple linear layout optimized for sequential access during playback. The file begins with a header encoding metadata such as frame dimensions, frame rate, total duration, and the positions of key frames. Following the header, the data packets and key frames appear in strict temporal order. This organization permits a streaming decoder to simply read forward through the file, applying each change event or key frame as it appears without requiring random access or complex seeking operations. The simplicity of the storage format reduces decoder logic complexity while maximizing throughput.

Decoder and Real-Time Reconstruction

The decoder architecture implements real-time video reconstruction from the sparse event representation through neuronal network-based prediction. Upon system boot with a compressed neuromorphic file loaded into memory, the decoder initializes by loading the first key frame into the frame buffer and preparing the neuronal networks for predictive computation. The neuronal weights have been pre-trained during encoding to understand the statistical structure of the video content, enabling accurate prediction of intermediate pixel values between key frames based on local context.

Playback proceeds by streaming through the compressed representation chronologically. Each change event packet is parsed to extract its temporal position, spatial location, and modification parameters. The neuronal substrate generates predictions for pixel values in regions not explicitly specified by change events, drawing upon patterns learned during encoding. These predictions fill in the spaces between known values, reconstructing a complete frame from the sparse representation. The predicted frame is written to SDRAM frame buffer and subsequently transmitted through video output pathways to display hardware.

The critical performance requirement is maintaining real-time frame rates of sixty frames per second to ensure smooth playback without visible stuttering. This demands that the reconstruction pipeline completes all processing for each frame within approximately sixteen milliseconds. The hardware architecture achieves this through parallelization where multiple neuronal computation pipelines operate simultaneously on different regions of the frame. The high bandwidth of SDRAM memory enables rapid reading of change event data and writing of reconstructed frames without bottlenecking the processing flow.

One fascinating capability emerges from the neuronal plasticity mechanisms that can remain active during playback. When a particular video is accessed frequently, the neuronal weights adapt to represent that specific content with increasing efficiency. The network essentially memorizes frequently accessed videos, enabling faster reconstruction and potentially higher quality playback for popular content. This neuronal caching effect provides automatic optimization without requiring explicit cache management logic, as the biological learning mechanisms naturally strengthen representations of repeatedly encountered patterns.

The video output stages transform the reconstructed frame buffer data into standard display signals. For VGA output, the resistor ladder Digital-to-Analog converter translates digital RGB values into analog voltage levels appropriate for legacy displays. Horizontal and vertical sync pulses are generated with precise timing to comply with standard video modes, typically 640x480 at sixty hertz refresh. The HDMI output employs dedicated transmitter chips to serialize the parallel pixel data into high-speed differential signals conforming to HDMI protocol specifications. Both outputs can operate simultaneously if desired, enabling mirrored display on multiple monitors.

Quality assessment of the reconstructed video requires comparison against the source material to quantify compression artifacts and information loss. Metrics such as peak signal-to-noise ratio and structural similarity index provide objective measures of reconstruction fidelity. The compression ratio is simply the original file size divided by the compressed representation size, with target ratios exceeding fifty-to-one for high frame rate source material with significant temporal redundancy. The balance between compression ratio and quality can be adjusted through threshold parameters controlling event sparsity and key frame frequency.

Million Neuron Simulation Architecture

The question of whether the fifty-dollar hardware platform can simulate one million biological neurons initially appears absurd given the finite logic resources of the FPGA. However, careful analysis of computational requirements reveals this capability as not merely possible but practically achievable through time-multiplexed processing. The key insight recognizes that neuron count and logic element count address different aspects of the simulation, with state storage capacity and update frequency as the actual constraints.

Each Izhikevich neuron requires storage for two sixteen-bit state variables representing voltage and recovery. This totals four bytes per neuron. One million neurons therefore require four megabytes of state storage, well within the thirty-two megabyte capacity of the onboard SDRAM with twenty-eight megabytes remaining for synaptic weights and frame buffers. The memory constraint thus poses no barrier to million-neuron simulations.

The logic constraint similarly proves surmountable through pipelining. A single arithmetic pipeline implementing the Izhikevich update equations can process any number of neurons sequentially by loading state from memory, executing the computation, and writing updated state back. The logic element count required for one such pipeline approximates two thousand elements, leaving over thirteen thousand elements available for control logic, memory interfaces, and caching structures. The number of neurons simulated depends not on replicating the pipeline but on how many times it can iterate through the neural population per unit time.

The update frequency calculation demonstrates feasibility. Assuming ten clock cycles per neuron including memory access latency, one million neurons require ten million cycles per complete update. At fifty megahertz clock frequency, this permits five complete updates per second. While biological cortex operates in the thirty to eighty hertz gamma rhythm range, the five hertz update rate achieved on budget hardware remains remarkable and sufficient for many neuromorphic computing applications. The simulation runs in slow motion relative to biology but captures the essential dynamics.

Optimization opportunities exist to improve update rates. Increasing parallelism through multiple arithmetic pipelines operating simultaneously can multiply throughput at the cost of additional logic elements. Eight parallel pipelines would achieve forty hertz update rates, approaching biological speeds. Careful memory interface optimization through burst reading and write buffering can reduce the cycles per neuron below ten. Operating the SDRAM at its maximum supported frequency rather than conservative settings increases memory throughput. These refinements can potentially achieve biological-timescale simulation with further engineering effort.

The synaptic connectivity challenge proves more severe than neuron state storage. One million neurons in a fully connected topology would require one trillion synaptic weights, vastly exceeding available storage. Even sparse connectivity with one hundred connections per neuron yields one hundred million synapses requiring two hundred megabytes of storage when using two bytes per weight. This exceeds SDRAM capacity and necessitates streaming weights from SD card or employing compressed connectivity rules such as small-world network topologies defined algorithmically rather than stored explicitly.

The mega pipeline architecture addresses these challenges through hierarchical memory organization. The SDRAM controller performs burst reads fetching batches of two hundred fifty-six neurons at once into cache implemented in the FPGA's embedded M9K memory blocks. The cache provides ultra-fast access for the computation core while the SDRAM controller prefetches the next batch in parallel. Eight parallel arithmetic pipelines process neurons from the cache simultaneously, maximizing utilization of the available logic resources. This organization balances memory bandwidth, computation throughput, and storage capacity to achieve million-neuron simulation on budget hardware.

Limitations and Known Issues Currently Being Solved

Several limitations and simplifications exist in the current implementation that warrant explicit acknowledgment. The network uses only excitatory connections. Biological cortex contains approximately 20 percent inhibitory interneurons that shape network dynamics through precisely timed suppression. Adding inhibitory neurons would improve stability and enable more complex computational patterns but would require additional logic resources and more sophisticated weight matrix structure.

Synaptic plasticity occurs offline. The weights are pre-calculated and loaded into memory before network operation. True neuromorphic systems implement spike-timing-dependent plasticity in hardware, allowing weights to adapt continuously based on observed spike patterns. The current architecture includes placeholder logic for STDP computation but this remains unimplemented pending future development.

The pattern completion task uses only a single learned pattern. Testing with multiple stored patterns would reveal capacity limits and interference effects inherent in attractor network dynamics. Preliminary testing with two patterns shows successful retrieval but with reduced robustness compared to single-pattern operation.

Timing closure at higher clock frequencies is expected to be challenging. The initial design target is 100 megahertz operation, but early synthesis reports and back-of-the-envelope path estimates suggest that critical paths in the multiplier chain will likely exceed 10 nanoseconds without additional pipelining. For this reason, the current NonPU design is scoped around a more conservative 50 megahertz clock, which should be attainable once the full top-level build compiles cleanly. Further pipeline optimization and restructuring of the arithmetic datapaths may allow higher operating frequencies in future revisions.

The VGA output provides only binary visualization. Adding grayscale rendering based on membrane voltage or firing rate would give more informative feedback about network state but would require additional video memory and more complex rendering logic.

The Neuronal Processing Unit (NonPU) is Being Developed Toward My Next Long-Term Research Vision

The NonPU project serves as foundational engineering work toward a broader research goal: creating interfaces between silicon computation and biological neuronal tissue. The skills developed here, including implementing spiking neural networks, understanding spike-timing dynamics, managing real-time signal processing under tight resource constraints, and designing hardware that speaks the language of spikes rather than floating-point matrices, are directly transferable to brain-computer interface and organoid intelligence research.

The connection is not arbitrary. A system designed to process millions of neuronal spikes per second, encode information in temporal patterns rather than continuous values, and maintain homeostatic stability through feedback control shares fundamental operational principles with the hardware required to interface with living neurons. The AC415 platform running the NonPU architecture could, with appropriate analog front-end circuitry, interpret electrical signals from electrode arrays contacting cultured neurons or brain organoids. The same differential encoding that compresses video by tracking what changes rather than what stays static could compress neural recordings by capturing spike events rather than continuous waveforms.

The goal is to make computing more efficient, less mechanical, and more fluid through the integration of Brain-Computer Interfaces and Organoid Intelligence. I plan to begin that research after achieving meaningful results here, specifically, a production-ready implementation of the Neuronal Processing Unit operating at least at 500 MHz, which will require transitioning to larger FPGA platforms such as the Cyclone V-based DE10-Nano with 110,000 logic elements and onboard DDR3 memory.
